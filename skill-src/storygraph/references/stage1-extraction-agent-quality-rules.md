# Stage 1 抽取 Agent 附加质量规则

以下规则必须在每次分派正文 `comprehensive_extraction` 抽取任务时一并发给抽取 agent。它们来自已完成产物的抽样审查问题，用于避免继续批量产生同类错误。

## 1. 只抽当前 chunk 明示的信息

- 不要把后文、前文或常识中的身份回填到当前 chunk。
- 当前 chunk 只写“田掌柜”“焉儿”“那两个帮会”时，就只能按当前称呼抽取；不能改成“田卜离”“燕焉”“五色门或独霸山庄”。
- 如果你知道后文身份，但当前 chunk 没明说，应放入 `uncertainties`，不要写成已验证节点或关系。
- 可以保留模糊称谓，例如“田掌柜”“五色门主之子”“某小头目”“青年男女修士”，但不要擅自补全姓名、门派或真实身份。

## 2. 路人角色默认不抽，除非有剧情功能

不要因为原文出现了人物称谓就机械建节点。以下低戏份角色默认不抽：

- 某个卖货的、店小二、普通伙计、路边摊主、普通守卫、普通仆役、普通车夫。
- 阿婆、老头、老妇、青年、少女、汉子等只承担背景说明或一次性问答的人物。
- 没有名字、没有决策、没有资源流转、没有冲突行为、没有关系影响、没有后续悬念的人物。

但以下情况需要抽取，即使表面像路人：

- 该角色推动了交易、冲突、追踪、情报传播、资源交付、陷阱、伪装或身份揭示。
- 该角色持有关键物品、提供关键情报、改变主角路线或引出重要势力。
- 该角色行为异常，可能是伪装、探子、高手、妖兽化形、敌对势力暗线。
- 该角色虽无名，但在当前 chunk 内产生明确事件后果，例如“黑衣老者拦截韩立”“店小二暗中报信”。

抽取这类角色时，优先使用当前文本称呼作为 label，并在 `uncertainties` 说明身份未明。

## 3. Evidence range 必须真正支撑 fact_summary

- `source_range` 不能只保证落在 packet 范围内，还必须覆盖能支撑 `fact_summary` 的原文片段。
- 不要用半句、断句或偏移错位的短 span 支撑长摘要。
- 如果摘要包含多个事实，证据范围必须覆盖这些事实；否则拆成多条 evidence，或缩短摘要。
- 遇到 CRLF/LF 差异时，按 task packet 的 LF 口径换算，但最终要人工/脚本确认 range 映射回的文本确实支撑事实。
- 不要让 evidence 的 `source_range` 指向相邻句、前后文或只含关键词但缺少动作/对象的片段。

## 4. 节点、边、事件引用必须闭合

- edge 的 source/target 必须能在 `extracted_nodes` 中找到。
- event 的 participants 必须能在 `extracted_nodes` 中找到。
- node/edge/event 引用的 evidence id 必须能在 `extracted_evidence` 中找到。
- 如果关系需要“墨府夫人”“燕歌”“四平帮小头目”“小楼屋内”等参与方，就必须建对应节点；否则不要写这条关系。
- 不要用自然语言名称作为 endpoint，必须用本文件内真实 node id。

## 5. 模板引用必须使用真实 requirement_id

- `supports_templates.template_name` 必须来自 `requirements/template-requirements.json`。
- `supports_templates.requirement_id` 必须对应真实 required field，不能自己造字段名。
- 例如不要写不存在的字段：`冲突过程`、`资源变化`、`功法名称`、`妖兽名称`、`符箓效果`、`进入条件`，除非 requirements 中确实存在。
- 如果不确定字段名，少写模板支持也比写错强。
- 模板支持应表达“这个对象/事件/证据覆盖了哪个模板需求”，不要把所有相关模板都泛泛挂上。

## 6. 不要过度类型化或过度关系化

- 原文只说“绿皮口袋”，不能直接确认为 `beast_container`，除非当前 chunk 明示它装妖兽、放灵兽或用于控制妖兽。
- 原文只说某丹药/古方对灵兽有用，不能直接写成它专门作用于某个具体妖兽卵，除非当前 chunk 明说。
- 对“某人背后有人指使”“那两个帮会”“某修士”等模糊信息，应保持模糊关系，不要自动填成已知势力或具体人物。
- 对推测型判断，优先写入 `uncertainties` 或 `rejected_candidates`。

## 7. 覆盖重点按剧情功能优先

优先抽取：

- 主要人物、明确有戏份的配角、当前 chunk 中做出决策或造成后果的人物。
- 势力、地点、法宝、丹药、功法、妖兽、材料、阵法、秘境、规则限制。
- 交易、冲突、追踪、逃遁、修炼、突破、死亡、夺宝、结盟、背叛、情报传播等事件。
- 资源流转：谁给了谁、获得/消耗了什么、代价和后果是什么。
- 因果链：为什么发生、由谁推动、造成了什么后续状态。

可以不抽或降级到背景：

- 单纯烘托场景的人群。
- 没有名字、没有动作后果、没有后续悬念的一次性角色。
- 与模板需求无明显关系的普通描写。

## 8. 统一 node_type 本体（必须从下表取值）

`node_type` 必须从下面这份修仙世界观通用本体中选取，不要自由发明新类型，也不要把同一类东西用多个近义类型表示。一份本体服务全部模板，下游模板按 `node_type` 取节点，类型写乱会导致模板查不到。

| node_type | 含义 | 典型例子 |
| --- | --- | --- |
| `character` | 具名或有剧情功能的人物 | 韩立、墨大夫 |
| `faction` | 势力、宗门、家族、帮会 | 七玄门、黄枫谷 |
| `location` | 地点、城池、区域、秘境入口 | 嘉元城、乱星海 |
| `facility` | 建筑、设施、场所功能单元 | 落云宗藏经阁、坊市 |
| `realm` | 修炼境界、修为层次 | 炼气期、筑基期、金丹期 |
| `cultivation_method` | 功法、心法、修炼体系 | 长春功、大衍诀 |
| `technique` | 术法、神通、招式、秘术 | 眨眼剑法、大须弥神光 |
| `pill` | 具名丹药、丸、丹 | 筑基丹、黄龙丹、金髓丸、定颜丹 |
| `medicine` | 药液、药剂、药水、灵液（非丸/丹形态的成药） | 雪灵水、天火液 |
| `material` | 灵药、药材、矿石、炼制原料 | 千年灵乳、地心火 |
| `formula` | 丹方、配方、炼制图纸/规则 | 黄龙丹丹方 |
| `artifact` | 法宝、灵宝、本命法宝 | 青竹蜂云剑、掌天瓶、金光罩 |
| `weapon` | 凡俗或非法宝级武器 | 普通飞剑、长枪 |
| `tool` | 法器、辅助器物、容器 | 储物袋、传讯符笔 |
| `formation` | 阵法、禁制、大阵 | 困灵阵、五行大阵 |
| `inscription` | 符箓、符篆、铭文 | 金光符、净魔符 |
| `beast` | 妖兽、灵兽、异兽 | 铁背蛮牛、火蟒 |
| `secret_realm` | 秘境、遗迹、机缘空间 | 乱星海遗迹 |
| `event` | 事件、情节节点 | 升仙大会、墨府之变 |
| `conflict` | 冲突、战斗、对抗 | 韩立 vs 蓝衣人 |
| `opportunity` | 机缘、机会点 | 拾得小绿瓶 |
| `rule` | 世界规则、机制、限制 | 筑基成功率受灵根影响 |

如果遇到一个实体明显不属于上表任何一类，先放入 `uncertainties` 说明，不要临时造类型；常见的难归类对象（如"绿皮口袋"）按 Rule 6 保守处理。

## 9. 具名物品必抽（丹药 / 法宝 / 材料 / 功法 / 阵法 / 符箓）

本规则用于纠正以往对物品类实体召回不足的问题。物品类实体的判定标准与人物**不同**：人物默认不抽路人（Rule 2），但**具名物品默认必抽**。

必抽（即使信息不全）：

- 任何**具名**的丹药、药液、材料、功法、术法、神通、法宝、武器、法器、阵法、符箓、妖兽，即使当前 chunk 只出现名字、没有完整效果/来源/炼制信息。
- 名字再奇怪也要抽：`xx丸`、`xxx液`、`xx散`、`xx膏`、`xx丹`、`xx剑`、`xx瓶`、`xx罩`、`xx幡`、`xx珠` 等任意命名形态，只要语境表明它是某类物品实体，就建节点。
- 缺失的属性（效果、来源、丹方、副作用、适用境界等）**不要因此跳过整个实体**，而是先建节点，把"未知/待确认"写入该节点相关的 `uncertainties`。

不要抽（避免污染）：

- 没点明名字的泛指物，如"一颗丹药""那件法宝""某种灵药"——按 Rule 1/Rule 6 放 `uncertainties`，不要硬造具名节点。
- 通用计量物资本身，如"灵石""木料""清水"，除非当前 chunk 把它当作具名关键物品对待。

判定口径：先问"原文有没有给它一个专属名字 + 语境是否表明它是物品"。两者都成立就必抽，且按 Rule 8 选 `node_type`（丹/丸→`pill`，液/水→`medicine`，剑/瓶/罩等法宝→`artifact`，药材矿石→`material`）。

## 10. 同一实体的称呼归一（label 与 aliases）

为后续跨 chunk 合并打基础，物品和人物在**当前 chunk 内**的多个称呼必须收敛到一个节点：

- 一个实体在本 chunk 有多个称呼时，选**最完整、最正式**的称呼作 `label`，其余全部写入 `aliases` 数组。例如本 chunk 同时出现"小绿瓶"和"掌天瓶"且原文明示是同一物，则 `label` 取一个、另一个进 `aliases`。
- `aliases` 只写**原文在本 chunk 明确指向同一实体**的称呼（包含简称、外号、别名、"也就是…""又称…"）。**不要**因为两个名字看起来像就归并，例如"金髓丸"与"金髓液"是不同实体，**不得**互相放入 aliases。
- 不确定两个称呼是否同一实体时，分别建节点，并在 `uncertainties` 说明"疑似同一实体，待确认"，由后续增量复核处理，不要在本步骤强行合并。
- `aliases` 是数组，元素为字符串；没有别名时写空数组 `[]` 或省略，不要写 `null`。

跨 chunk 的同名/别名最终合并由 merge 阶段处理；抽取 agent 只需保证**本 chunk 内**称呼已正确归一并如实填写 `aliases`。

## 11. 输出前自检清单

每个 chunk 写盘前必须自检：

- JSON 可解析，UTF-8 without BOM。
- 固定字段正确：`run_id=run-001`、`lane_id=comprehensive_extraction`、`agent_role=comprehensive-stage1-extraction-agent`、`model_or_agent_identity=codex-subagent`、`output_status=completed`。
- 所有 node/edge/event/evidence 都有合法 `confidence` 和 `verification_status`；`confidence` 只能使用配置枚举，例如 `EXTRACTED`、`INFERRED`、`AMBIGUOUS`，不要写 `high`、`medium`、`extracted` 等临时值。
- 所有 node/edge/event/evidence 都有可验证的 source locator：优先同时写 `source_range` 和对象型 `source_location`，例如 `{"chunk_id":"chunk-0001","source_range":[0,100]}`；不要只写字符串型 `source_location`。
- 所有 source_range 在 packet 范围内，且映射文本能支撑 fact_summary。
- 所有 node/edge/event 引用的 evidence id 必须存在于 `extracted_evidence`；edge 的 source/target、event 的 participants 必须使用本文件内真实 node id，不能使用自然语言名称。
- 所有 template_name 和 requirement_id 真实存在。若最终 `requirements/template-requirements.json` 是 summary 结构，`requirement_id` 使用真实 summary category id；不要留空或临时编造字段名。
- 没有本 chunk 未明示的姓名、势力、身份、物品功能或关系。
- 低戏份路人没有被无意义建节点；疑似伪装或关键路人已用当前称呼保守抽取并写明不确定性。
- 所有 `node_type` 都取自 Rule 8 的本体表，没有自由发明的类型；难归类对象已放 `uncertainties`。
- 本 chunk 出现的具名丹药/药液/材料/功法/术法/法宝/武器/法器/阵法/符箓/妖兽都已按 Rule 9 建节点，未因信息不全而漏抽；泛指物未被硬造成具名节点。
- 同一实体在本 chunk 的多个称呼已按 Rule 10 归一：选最完整称呼作 `label`，其余进 `aliases`；未把不同实体（如"金髓丸"与"金髓液"）错误归并。
